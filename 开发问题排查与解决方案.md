# GovRAG 开发问题排查与解决方案汇总

## 阶段 0：基础设施搭建

### 问题 1：MySQL 连接拒绝（`localhost` vs `mysql`）

**现象**：Flask 启动后 `/health` 接口报错 `Can't connect to MySQL server on 'mysql'`。

**根因**：`.env` 中 `DB_HOST=mysql` 是 Docker 容器网络内的 DNS 名，Flask 在宿主机运行无法解析。

**解决**：`.env` 中改为 `DB_HOST=127.0.0.1`，端口用 docker-compose 映射的宿主机端口 `DB_PORT=3307`（后切换为本地 MySQL 后改为 `3306`）。

```env
# 正确配置（Flask 在宿主机，MySQL 在 Docker）
DB_HOST=127.0.0.1
DB_PORT=3307

# 正确配置（Flask 和 MySQL 都在宿主机）
DB_HOST=127.0.0.1
DB_PORT=3306
```

### 问题 2：Infinity SDK 在 Windows 上崩溃

**现象**：`import infinity` 抛出 `cannot convert longdouble infinity to integer`。

**根因**：`infinity-sdk` 的 C++ 扩展文件（`.pyd`）与 **Python 3.13/3.14** 不兼容。Python 3.13+ 内置了 `infinity` 浮点常量符号，与 SDK 的 C++ 命名空间冲突。

**解决**：1. 健康检查改用 HTTP 直连替代 SDK；2. 使用 Python 3.12 环境可正常导入。

```python
# 修复前（SDK，Windows 崩溃）
import infinity
client = infinity.connect(uri)

# 修复后（HTTP 直连，绕过 SDK C++ 层）
import requests
resp = requests.get(f"http://{host}:{port}/admin/node/status", timeout=3)
```

### 问题 3：放弃 Docker 后的本地服务管理

**现象**：放弃 Docker 后 MyQL/Redis/MinIO 全部需要在本地启动，手动管理多个终端窗口。

**根因**：RAGFlow 原项目用 `docker compose up -d` 一键拉起 6 个容器。我们选择本地部署后每个中间件都是独立进程。

**解决**：MySQL 用 Windows 服务自启动，Redis 和 MinIO 各自一个终端常驻。MinIO 下载安装步骤：

```powershell
# MinIO 下载
mkdir D:\9_1_Enviroment\MinIO -Force
Invoke-WebRequest -Uri "https://dl.min.io/server/minio/release/windows-amd64/minio.exe" `
  -OutFile "D:\9_1_Enviroment\MinIO\minio.exe"

# MinIO 启动（新终端常驻）
$env:MINIO_ROOT_USER="minioadmin"
$env:MINIO_ROOT_PASSWORD="minioadmin"
D:\9_1_Enviroment\MinIO\minio.exe server D:\9_1_Enviroment\MinIO\data --console-address ":9001"
```

## 阶段 1：鉴权闭环

### 问题 4：MySQL 没有原生 UUID 类型

**现象**：SQLAlchemy 模型定义中 `db.Column(UUID(as_uuid=True))` 在 MySQL 下无法工作。

**根因**：`UUID` 类型来自 `sqlalchemy.dialects.postgresql`，是 PostgreSQL 独有的。MySQL 没有原生 UUID 类型。

**解决**：自定义 `GUID` 类型装饰器，MySQL 侧存 `CHAR(36)`，Python 侧自动转换 `uuid.UUID`。

```python
from sqlalchemy.types import TypeDecorator, CHAR

class GUID(TypeDecorator):
    impl = CHAR(36)
    cache_ok = True

    def process_bind_param(self, value, dialect):
        return str(value) if value else None

    def process_result_value(self, value, dialect):
        return uuid.UUID(value) if value else None

# 使用
id = db.Column(GUID(), primary_key=True, default=uuid.uuid4)
```

### 问题 5：登录锁定时间计算 bug

**现象**：锁定时间计算使用了两次 `datetime.now()`，可能跨秒产生错误。

**根因**：
```python
# 错误写法
user.locked_until = datetime.now(timezone.utc).replace(
    minute=datetime.now(timezone.utc).minute + 30  # 两次调用不在同一时刻
)
```

**解决**：用 `timedelta` 替代 `replace`：
```python
from datetime import timedelta
user.locked_until = datetime.now(timezone.utc) + timedelta(minutes=30)
```

### 问题 6：文件命名未对标 RAGFlow 源项目

**现象**：`api/apps/auth/` 下有 `login.py` 和 `middleware.py`，但 RAGFlow 原项目 `auth/` 下没有这两个文件。

**根因**：RAGFlow 采用"垂直模块聚合"设计——鉴权装饰器 `login_required` 和登录路由全部在 `api/apps/__init__.py` 单体文件中（15KB），`api/apps/auth/` 只放 OAuth 提供者。

**解决**：将 `login.py`、`middleware.py` 合并进 `api/apps/__init__.py`，删除拆分文件，装饰器命名为 `login_required` 对标原项目。

```
修复前                            修复后
──────────────────────────────────────────────────
api/apps/auth/__init__.py         api/apps/__init__.py  ← auth_bp + login_required + 登录路由
api/apps/auth/login.py            api/apps/auth/__init__.py  ← 空文件（预留给 OAuth）
api/apps/auth/middleware.py
```

## 阶段 2：数据管理

### 问题 7：文件上传→文档创建链路断开

**现象**：上传文件到 MinIO 后，创建文档时需要手动填写 `file_path`、`file_type` 等字段，与已上传的文件完全没有关联。

**根因**：`POST /files/upload` 返回 `File` 记录，`POST /datasets/:id/documents` 创建 `Document` 记录——两个接口之间没有任何数据桥接。

**解决**：`create_document()` 改为接收 `file_ids` 数组，自动从 `File` 记录填充 `name`、`file_type`、`file_size`、`file_path`。

```python
# 修复后
@document_bp.route("/datasets/<dataset_id>/documents", methods=["POST"])
def create_document(dataset_id):
    file_ids = data.get("file_ids", [])
    files = File.query.filter(File.id.in_(file_ids), File.tenant_id == g.tenant_id).all()
    for f in files:
        doc = Document(
            name=f.original_name,      # ← 自动填充
            file_type=f.mime_type,      # ← 自动填充
            file_size=f.file_size,      # ← 自动填充
            file_path=f.object_key,     # ← 自动填充
            ...
        )
```

### 问题 8：删除数据集不删除文档

**现象**：`DELETE /datasets/:id` 只删了 Dataset 行，关联的 Document 成为孤儿记录。

**根因**：`Document.dataset_id` 没有声明 `db.ForeignKey("datasets.id")`，模型也没有定义 `db.relationship`。

**解决**：
```python
# db_models.py — Document 模型
dataset_id = db.Column(GUID(), db.ForeignKey("datasets.id"), nullable=False, index=True)

# Dataset 模型加级联关系
documents = db.relationship("Document", backref="dataset", cascade="all, delete-orphan")
```

---

### 问题 9：删除文件时 MinIO 对象残留

**现象**：`DELETE /files/:id` 只删了 DB 记录，MinIO 里 `govrag/xxx/yyy.pdf` 仍然占用存储空间。

**解决**：在 `file_api.delete_file()` 中先调用 `s3_client.remove_object()` 再删 DB 记录。

```python
def delete_file(file_id):
    db_file = File.query.filter_by(id=file_id, tenant_id=g.tenant_id).first()
    remove_object(db_file.object_key, db_file.bucket_name)  # ← 先清理 MinIO
    db.session.delete(db_file)
    db.session.commit()
```

### 问题 10：`document_count` 手动计数器并发不安全

**现象**：创建/删除文档时手动 `+1`/`-1`，并发请求会导致计数错误。

**解决**：改为 `db.session.flush()` 后实时 count 查询，不再手动维护计数器。

```python
# 修复前
dataset.document_count = (dataset.document_count or 0) + 1

# 修复后
db.session.flush()
dataset.document_count = Document.query.filter_by(dataset_id=...).count()
```

## 阶段 3：文档解析流水线

### 问题 11：Python 3.12 vs 3.14 —— `infinity-sdk` 导入差异

**现象**：Python 3.13/3.14 下 `import infinity` 崩溃，Python 3.12 下正常。

**根因**：Python 3.13+ 内置了 `infinity` 浮点常量，与 `infinity-sdk` 的 C++ 命名冲突。Python 3.12 没有该符号，因此兼容。

**解决**：使用 Python 3.12 的 conda 环境（`omni`）进行开发。生产环境用 Linux 运行时不存在此问题。

---

### 问题 12：`faiss.search()` 索引越界

**现象**：`results.append((self._id_list[idx], float(dist)))` 抛出 `IndexError: list index out of range`。

**根因**：`add()` 方法中 `self._id_list` 只有一行裸引用，没有调用 `.extend(chunk_ids)`，导致 `_id_list` 始终为空列表。

```python
# 错误代码（第 26 行）
self.index.add(vectors.astype(np.float32))
self._id_list          # ← 什么都没做！只是引用了变量

# 正确代码
self.index.add(vectors.astype(np.float32))
self._id_list.extend(chunk_ids)
```

另外还发现 `__init__` 中加载时写的是 `self.id_list`（无下划线），搜索时读的是 `self._id_list`（有下划线）——命名不一致导致重载后数据丢失。

---

### 问题 13：重复运行导致 FAISS 向量重复叠加

**现象**：同一个测试脚本多次运行后 `ntotal=12`，`_id_list=['a','b','c','a','b','c',...]`。

**根因**：`__init__` 从磁盘加载已有索引后，再次 `add()` 会追加而非覆盖。

**解决**：这是设计行为——生产环境中每个数据集只解析一次文档，不会重复。测试时每次删 `test.index/` 目录即可。

---

### 问题 14：4 个 `__init__.py` 缺失导致模块导入失败

**现象**：`from deepdoc.parser.pdf_parser import PdfParser` 抛出 `ModuleNotFoundError`。

**根因**：Python 需要 `__init__.py` 将目录识别为包。以下文件缺失：
- `backend/deepdoc/parser/__init__.py`
- `backend/deepdoc/chunker/__init__.py`
- `backend/rag/__init__.py`
- `backend/common/__init__.py`

**解决**：全部创建空文件。

### 问题 15：`naive_merge()` 第一个 chunk 为空串

**现象**：输出 `Chunk0: tokens=0, text=`。

**根因**：RAGFlow 原代码初始化 `cks = [""]`，第一次 `add_chunk` 因 `cks[-1] == ""` 条件触发后 `append` 新内容，空串 `""` 留在位置 0 成为垃圾。

**解决**：返回前过滤空字符串：

```python
return [ck for ck in cks if ck]
```

---

### 问题 16：Celery Worker 找不到任务函数

**现象**：`Received unregistered task of type 'api.tasks.document_tasks.parse_document'`。

**根因**：Celery 启动时没有加载 `document_tasks.py` 模块，任务未被注册。

**解决**：使用 `imports` 配置显式导入，而非依赖 `autodiscover_tasks` 自动扫描：

```python
# celery_app.py
celery.conf.update(
    imports=['api.tasks.document_tasks'],
)
```

---

### 问题 17：Celery Worker 抛出 `Working outside of application context`

**现象**：Worker 执行 `db.session.commit()` 时抛 Flask 上下文异常。

**根因**：Celery Worker 是独立进程，没有 HTTP 请求，Flask 的 `app_context` 和 `db.session` 均未初始化。

**解决**：任务函数内手动创建 Flask 应用上下文：

```python
@celery.task(bind=True, max_retries=3)
def parse_document(self, doc_id: str):
    from api import create_app
    app = create_app()
    with app.app_context():
        doc = Document.query.get(doc_id)
        # ... 所有数据库操作
```
