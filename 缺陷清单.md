# 阶段一
## 流程
简要总结下该阶段实现逻辑，并校对是否存在缺陷。梳理流程 + 缺陷审计：

POST /api/v1/auth/login  {"username":"admin","password":"admin123"}
│
├── login.py:login()
│   ├── User.query.filter_by(username="admin").first()          → 找到用户
│   ├── user.verify_password("admin123")                        → bcrypt 校验通过
│   ├── user.login_failed_count = 0, user.locked_until = None   → 重置锁定状态
│   ├── user.last_login_at = now()                              → 记录登录时间
│   ├── db.session.commit()
│   ├── jwt_utils.create_access_token(user_id, tenant_id, ...)  → HS256 JWT，2h 有效
│   └── jwt_utils.create_refresh_token(user_id)                 → refresh token，7d 有效
│
└── 返回 {"code":0, "data": {"token":"eyJ...", "refresh_token":"eyJ...", "user":{...}}}

GET /api/v1/auth/me  Authorization: Bearer eyJ...
│
├── middleware.py:require_auth()
│   ├── _extract_token() → 从 Header 取 Bearer token
│   ├── jwt_utils.decode_token(token) → HS256 验签+过期检查
│   ├── payload["type"] != "refresh"  → 拒绝 refresh token 当 access token 用
│   ├── User.query.get(payload["sub"]) → 查用户是否存在且 is_active
│   └── 注入 g.current_user, g.tenant_id, g.security_level, g.role
│
└── login.py:get_current_user() → return g.current_user.to_dict()

## 缺陷清单
🔴 需要修的
1. login.py 锁定时间计算有 bug
user.locked_until = datetime.now(timezone.utc).replace(
    minute=datetime.now(timezone.utc).minute + 30
)
两次 datetime.now() 不是同一时刻。第二次调用的秒数可能比第一次小，导致 minute + 30 跨小时后秒数错乱。应该改成：

from datetime import timedelta
user.locked_until = datetime.now(timezone.utc) + timedelta(minutes=30)
🟡 目前不影响，但未来要补
2. 没有 refresh token 刷新接口

当前 login 返回了 refresh_token，但 api/apps/auth/login.py 里没有 POST /refresh-token 端点。前端如果 access token 过期了，只能重新登录。影响不大（2h 够用），但阶段 3 做前端时得补上。

3. 没有登出 / Token 失效机制

登出只能靠前端删 localStorage，服务端没有黑名单。Token 被盗用后无法主动撤销。政务场景需要补 POST /logout + Redis 黑名单。

4. middleware 没有做租户隔离校验

require_auth 把 tenant_id 注入 g 了，但没有校验请求参数里的 tenant_id 是否匹配 JWT 中的。比如用户 A（租户1）可以传 tenant_id=租户2 去查租户2的数据集——只要数据集接口没有自己做过滤。

5. HS256 对称密钥

JWT 用的是 HS256（同一个密钥签发+验签），适合单服务。如果将来会有多个微服务验签，需要切 RS256。当前 Flask 单体没问题。

# 阶段二
安装miniio 创建桶 govrag

数据集 CRUD：
  POST   /api/v1/datasets              → Dataset(name, embedding_model, tenant_id) → commit
  GET    /api/v1/datasets              → 按 tenant_id 分页查询
  GET    /api/v1/datasets/:id          → 单条查询（租户隔离）
  PUT    /api/v1/datasets/:id          → 部分更新
  DELETE /api/v1/datasets/:id          → 直接删除（⚠️ 级联问题见下方）

文件上传：
  POST   /api/v1/files/upload          → secure_filename → s3_client.upload_file()
                                         → MinIO put_object → File() → commit
  GET    /api/v1/files                 → 按 tenant_id 分页列表
  DELETE /api/v1/files/:id             → 删 DB 记录（⚠️ MinIO 文件未删）

文档管理：
  POST   /api/v1/datasets/:id/documents → 验证 Dataset 存在 → Document() → document_count+1
  GET    /api/v1/datasets/:id/documents → 分页列表
  GET    /api/v1/documents/:id          → 单条
  DELETE /api/v1/documents/:id          → document_count-1 → 删除

🔴 严重——功能不完整
1. 文件上传和文档创建是断开的

POST /files/upload     → File 存入 MinIO + File 表记录
POST /datasets/:id/documents → 创建 Document 记录
                                 ↑
                          没有任何关联 File
Document 模型有 file_path 字段，但 create_document() 只是从 JSON body 读 file_path，前端传什么就是什么——跟刚才上传的 File 没有任何关系。RAGFlow 的做法是上传文件时同时创建 Document，一步到位。

修复：POST /datasets/:id/documents 应该接收 file_ids 列表，自动根据 File 记录填充 name、file_type、file_size、file_path。

🟡 中等——会影响后续阶段
2. 删除数据集没有级联删除文档

Dataset 和 Document 之间没有在模型层定义 db.relationship，所以删数据集时 MySQL 不知道要同时删文档。delete_dataset() 只删了 Dataset 本身，文档和文件记录变成孤儿。

修复：在 Dataset 模型上加 documents = db.relationship("Document", backref="dataset", cascade="all, delete-orphan")。

3. 删除文件时 MinIO 里的对象没删

delete_file() 只删了 DB 里的 File 记录，MinIO 里 govrag/xxx/yyy.pdf 还在。时间长了存储空间只增不减。

修复：delete_file() 里先调 s3_client.remove_object(object_key) 再删 DB 记录。

4. document_count 计数器不可靠

dataset.document_count = (dataset.document_count or 0) + 1
如果两个请求并发创建/删除文档，计数会出错。MySQL 默认事务隔离级别下，READ COMMITTED 会有丢失更新问题。

修复：阶段 5 时用 db.session.query(Dataset).filter_by(id=...).update({"document_count": Dataset.document_count + 1}) 原子更新；或者放弃手动维护计数器，改为用 Document.query.filter_by(dataset_id=...).count() 实时查询。

🟢 轻微——当前不阻塞
5. 没有文件类型和大小校验

file_api.py 接受任何 MIME 类型和大小，上传一个 2GB 的视频也能通过。

6. Dataset 接口缺少 security_level 过滤

list_datasets() 按 tenant_id 过滤了，但没有按 security_level <= g.security_level 过滤。虽然当前没有涉密数据，但阶段 5 加了这个字段后接口就漏了。

7. uploaded_by 字段在 Document 中没有自动赋值

create_document() 手动设了 uploaded_by=g.current_user.id，但如果前端传了 uploaded_by 字段（JSON body），会覆盖掉——应该强制用后端值。
